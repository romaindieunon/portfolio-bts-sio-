Projet 2 — PFsense (Pare-feu & VLAN)

1. Contexte et objectifs

Ce second projet de l'épreuve E6 vient compléter le Projet 1 (Active Directory) : il met en place la couche réseau et sécurité qui protège l'infrastructure et permet aux clients d'accéder à Internet. Le cahier des charges fixait trois objectifs principaux :

  • Sécuriser la sortie Internet et les flux entre les sous-réseaux par un pare-feu open-source.
  • Segmenter le réseau en deux VLAN (serveur et client) pour isoler les ressources sensibles.
  • Garantir la continuité de service par la mise en place de deux PFsense en haute disponibilité (CARP).

2. Architecture réseau cible

L'infrastructure complète (pare-feu, switch, serveurs, clients) est représentée ci-dessous (schéma commun aux deux projets de l'épreuve E6) :

Détail des éléments représentés :

  • Routeur FAI — sortie Internet (deux WAN en IP auto sur Alpha et Lambda).
  • Alpha — PFsense principal : LAN 192.168.100.250, VLAN 200 192.168.200.250, VLAN 201 192.168.201.250.
  • Lambda — PFsense de secours : LAN 192.168.100.251. Reprend l'IP virtuelle en cas de panne.
  • IP virtuelle (CARP) — 192.168.100.1, point d'entrée unique vu des clients.
  • Switch Theta — Fa0/1-12 (VLAN 200), Fa0/13-22 (VLAN 201), Fa0/23-24 + Gi0/1-2 (trunks vers les PFsense).
  • VLAN 200 — Serveurs (192.168.200.0/24) — Beta, Delta (AD/DNS/DHCP), Epsilon (GLPI), Kappa (TrueNAS), Gamma (Proxmox).
  • VLAN 201 — Clients (192.168.201.0/24) — postes ZET-xx + borne Wi-Fi ETA, IP attribuées par DHCP.

Les deux PFsense se synchronisent en temps réel par XMLRPC sync : toute modification faite sur Alpha (règles, VLAN, NAT, alias) est immédiatement répliquée sur Lambda.

3. Environnement technique

  • Hyperviseur : VMware Workstation (machines virtuelles avec deux cartes réseau).
  • Système : PFsense CE 2.7 (basé sur FreeBSD).
  • Switch : Cisco Catalyst (Theta) — VLAN dot1Q avec trunks.
  • Cartes réseau par PFsense : WAN (sortie Internet) + LAN (interface VLAN trunk).
  • Outils : console série, navigateur web (interface d'administration HTTPS).
1

Préparation et démarrage de l'installation

PFsense est un pare-feu open-source basé sur FreeBSD. Je commence par récupérer l'image ISO officielle depuis pfsense.org et je crée une machine virtuelle dédiée avec deux cartes réseau (une pour le WAN sortie Internet, une pour le LAN côté infrastructure interne). Je démarre la VM sur l'ISO, accepte le contrat de licence et choisis l'option Install dans le menu de boot.

Installation du système PFsense

L'installateur propose plusieurs étapes : choix du clavier (fr-pc), sélection du disque cible, type de partitionnement (j'utilise Auto (UFS), GPT BIOS), et copie des fichiers système. Le processus de copie prend quelques minutes. À la fin, l'installateur propose de réinstaller depuis le shell ou de redémarrer : je choisis Reboot et retire l'ISO de la VM pour démarrer sur le disque.

3

Configuration initiale par la console (assignation des interfaces)

Après le redémarrage, PFsense s'ouvre sur un menu console. La première étape critique est l'assignation des interfaces réseau : l'option 1 demande d'identifier la carte WAN (sortie Internet) et la carte LAN (réseau interne) en utilisant les noms d'interfaces FreeBSD (em0, em1, etc.). L'option 2 permet ensuite d'attribuer une IP fixe sur le LAN — je configure 192.168.100.250 (Alpha) avec un masque /24, et je désactive le DHCP côté LAN car ce service est assuré par les serveurs Beta et Delta.

4

Accès à l'interface web et configuration générale

Une fois le LAN configuré, je peux accéder à PFsense depuis un navigateur sur l'IP du LAN (https://192.168.100.250). L'identifiant par défaut est admin / pfsense — la première connexion lance un assistant Setup Wizard qui demande le hostname (Alpha), le domaine (btssio.rd), les serveurs DNS (les contrôleurs de domaine Beta et Delta), le fuseau horaire et les paramètres WAN. Je termine en changeant le mot de passe administrateur par un mot de passe fort.

Captures — page de connexion, assistant Setup Wizard, paramètres généraux (8 captures) :

5

Création et configuration des VLAN (200 serveur, 201 client)

Pour segmenter le trafic, je configure deux VLAN sur l'interface LAN dans Interfaces → Assignments → VLANs. Je crée le VLAN 200 (réseau serveurs 192.168.200.0/24, IP de PFsense 192.168.200.250) et le VLAN 201 (réseau clients 192.168.201.0/24, IP de PFsense 192.168.201.250). Chaque VLAN est ensuite affecté en tant qu'interface dans Interfaces → Assignments, puis activé et nommé. Côté switch Theta, les ports correspondants sont configurés en access sur le bon VLAN, et les ports vers PFsense en trunk (dot1q).

6

Règles de pare-feu et NAT

Par défaut, PFsense bloque tout trafic entrant. Je configure dans Firewall → Rules les règles nécessaires : autorisation du trafic inter-VLAN contrôlé (les clients du VLAN 201 doivent joindre le DHCP/DNS du VLAN 200, mais pas accéder librement aux autres serveurs), autorisation du trafic sortant vers Internet (NAT automatique sur le WAN), blocage explicite des protocoles non utilisés. Je vérifie également les règles de NAT outbound pour s'assurer que les VLAN 200 et 201 sortent bien sur l'IP publique du WAN.

7

Installation et configuration du second PFsense (Lambda)

Pour la haute disponibilité, je répète l'installation sur une seconde machine, Lambda. Je lui attribue une IP fixe différente sur le LAN (192.168.100.251) et configure les VLAN avec des IP propres (par convention : .249 sur les deux VLAN pour Lambda, .250 restant l'IP de Alpha). Je m'assure que le hostname est unique et que les deux PFsense partagent les mêmes DNS et le même domaine.

8

Configuration CARP — haute disponibilité (HA)

Le mécanisme CARP (Common Address Redundancy Protocol) permet de partager une IP virtuelle entre deux PFsense : si le maître tombe, le secours prend immédiatement l'IP. Dans Firewall → Virtual IPs, je crée trois VIP CARP — une par interface (LAN 192.168.100.1, VLAN 200 et VLAN 201) — avec le même mot de passe et le même VHID sur les deux machines. Dans System → High Availability Sync, j'active la synchronisation XMLRPC entre Alpha (maître) et Lambda (secours) afin que toutes les règles, VLAN et configurations soient répliquées en temps réel. Lambda devient ainsi un miroir de Alpha.

Tests et vérifications

Je valide la mise en service en faisant plusieurs tests : ping vers Internet depuis un client du VLAN 201, ping inter-VLAN contrôlé, vérification du basculement en éteignant Alpha (Lambda doit reprendre l'IP virtuelle CARP en quelques secondes), vérification de la synchronisation des règles depuis l'interface Status → CARP, et redémarrage de Alpha pour vérifier qu'il reprend automatiquement le rôle de maître à son retour.

Bilan et compétences mobilisées

Ce projet PFsense vient compléter naturellement le Projet 1 (Active Directory) : ensemble, ils forment une infrastructure complète, sécurisée et résiliente. J'ai notamment consolidé les compétences suivantes du référentiel SISR :

  • Installer et configurer un pare-feu open-source (PFsense / FreeBSD) en environnement virtualisé.
  • Segmenter un réseau par VLAN (norme 802.1Q, trunks, sous-interfaces).
  • Définir une politique de filtrage (règles de pare-feu, NAT, contrôle inter-VLAN).
  • Mettre en place la haute disponibilité avec CARP et synchronisation XMLRPC.
  • Tester et documenter la solution (procédures, captures, scénarios de panne).

Les tests finaux ont validé : un client du VLAN 201 sort sur Internet via le NAT d'Alpha, Lambda reprend l'IP CARP 192.168.100.1 en quelques secondes lorsque Alpha est éteinte, la synchronisation XMLRPC est fonctionnelle (toute règle créée sur Alpha apparaît immédiatement sur Lambda) et le routage inter-VLAN respecte la politique définie.
