Projet 2 — PFsense (Pare-feu & VLAN)

1. Contexte et objectifs

Ce second projet de l'épreuve E6 vient compléter le Projet 1 (Active Directory) : il met en place la couche réseau et sécurité qui protège l'infrastructure et permet aux clients d'accéder à Internet. Le cahier des charges fixait quatre objectifs :

  • Sécuriser la sortie Internet et les flux entre les sous-réseaux par un pare-feu open-source.
  • Segmenter le réseau en deux VLAN (serveur et client) pour isoler les ressources sensibles.
  • Détecter les intrusions via un IDS/IPS Snort sur les interfaces critiques.
  • Garantir la continuité de service par deux PFsense en haute disponibilité (cluster CARP).

2. Architecture réseau cible

L'infrastructure complète (pare-feu, switch, serveurs, clients) est représentée ci-dessous (schéma commun aux deux projets de l'épreuve E6) :

  • Routeur FAI — sortie Internet (deux WAN en IP auto sur Alpha et Lambda).

Les pare-feu Alpha et Lambda (en surbrillance) sont au cœur de ce projet.

  • Alpha — PFsense principal : LAN 192.168.100.250, VLAN 200 192.168.200.250, VLAN 201 192.168.201.250.
  • Lambda — PFsense de secours : LAN 192.168.100.251. Reprend l'IP virtuelle en cas de panne.
  • IP virtuelle CARP — 192.168.100.1, point d'entrée unique vu des clients.
  • Switch Theta — Fa0/1-12 (VLAN 200), Fa0/13-22 (VLAN 201), Fa0/23-24 + Gi0/1-2 (trunks vers les PFsense).
  • VLAN 200 — Serveurs — Beta/Delta (AD, DNS, DHCP), Epsilon (GLPI), Kappa (TrueNAS), Gamma (Proxmox).
  • VLAN 201 — Clients — postes ZET-xx + borne Wi-Fi ETA, IP attribuées par DHCP relayé.

3. Environnement technique

  • Hyperviseur : Proxmox VE (sur le serveur Gamma — 192.168.200.211).
  • Système : PFsense CE dernière version stable (basé sur FreeBSD).
  • Switch : Cisco Catalyst (Theta) — VLAN 802.1Q avec trunks.
  • Cartes réseau par PFsense : WAN (sortie Internet) + LAN (interface VLAN trunk).
  • Outils : navigateur web (interface d'administration HTTPS), client SSH, console.
  • Modules : Snort (IDS/IPS), CARP (haute disponibilité).
1

Présentation et lancement de l'installation

PFsense est un pare-feu et routeur open-source basé sur FreeBSD. Il est hautement modulable et permet d'ajouter des extensions (plugins) selon les besoins de l'infrastructure. Cette procédure couvre son installation complète puis sa configuration en pare-feu, avec gestion des VLAN, du routage inter-VLAN, d'un IDS/IPS Snort et de la haute disponibilité par cluster CARP. Je commence par démarrer la machine virtuelle sur l'image ISO de PFsense Community Edition et accepter le contrat de licence.

Sélection des cartes réseau WAN et LAN

L'installateur demande d'identifier les cartes réseau. Je sélectionne d'abord la carte qui sera dédiée au WAN (sortie Internet) — j'ai vérifié au préalable les adresses MAC pour bien identifier l'interface correspondante. Le WAN est configuré en DHCP pour récupérer automatiquement une IP du FAI ; les paramètres par défaut sont conservés (modifiables ensuite). Je sélectionne ensuite la carte destinée au LAN, c'est-à-dire celle utilisée pour le réseau interne de l'entreprise.

Choix du système de fichiers, du disque et de la version

Une étape de souscription au support est proposée — je ne souscris pas car j'utilise la version gratuite. Vient ensuite le choix du système de fichiers et du schéma de partition (je conserve les paramètres par défaut, parfaitement adaptés). PFsense propose une option de redondance (RAID) que je laisse de côté car je n'ai qu'un seul disque dans la VM. Je sélectionne le disque cible, confirme la suppression de toutes les données présentes, puis choisis la dernière version stable de PFsense à installer.

Copie du système et redémarrage

L'installation se lance : copie des fichiers système sur le disque, configuration des binaires, mise en place du bootloader. Le processus prend quelques minutes. Une fois terminé, l'installateur propose de redémarrer ou d'ouvrir un shell. Je choisis Reboot et je retire l'ISO de la VM pour démarrer directement sur le disque.

5

Configuration de l'IP LAN via la console

Au redémarrage, PFsense affiche son menu console. Pour configurer l'IP LAN, je sélectionne l'option 2 : Set interface IP address. Je choisis l'interface LAN, puis je refuse l'activation du DHCP (le rôle DHCP du réseau client sera assuré ailleurs ou via relais). Je refuse aussi de repasser en HTTP afin de conserver le protocole HTTPS qui chiffre l'accès à l'interface web. À la fin, PFsense affiche l'URL de l'interface web sécurisée et je peux y accéder depuis un poste connecté au LAN.

Captures — menu console, option 2, configuration IP LAN, refus DHCP, validation HTTPS (4 captures, dans l'ordre du document) :

6

Setup Wizard : hostname, domaine, DNS, fuseau horaire

Une fois connecté à l'interface web, l'assistant de configuration se lance automatiquement. Je définis :

  • Hostname : Alpha (PFsense principal du cluster).
  • Nom de domaine : BTSSIO.RD, identique à mon domaine AD.
  • DNS primaire : 1.1.1.1 (Cloudflare) — DNS publics utilisés temporairement.
  • DNS secondaire : 9.9.9.9 (Quad9). Ces serveurs publics seront remplacés par les DC internes (Beta/Delta) une fois l'AD opérationnel.
  • Fuseau horaire : Europe/Paris (cohérence des journaux et horaires système).
Pour le WAN, je conserve le mode DHCP mais je décoche l'option Block private networks from entering via WAN pour autoriser le trafic depuis les plages RFC 1918 (utile dans notre lab où le "WAN" est en réalité un réseau privé en amont).

7

Finalisation : IP LAN définitive et mot de passe administrateur

Je termine l'assistant en définissant l'IP LAN définitive : 192.168.100.250/24 conformément au schéma réseau (LAN BTSSIO). Je dois alors me reconnecter sur cette nouvelle adresse et adapter l'IP du poste client dans la même plage. Pour des raisons de sécurité, je remplace le mot de passe administrateur par défaut par un mot de passe fort (≥ 15 caractères, majuscules + minuscules + chiffres + caractères spéciaux) afin de réduire les risques d'attaque par force brute.

Mises à jour, accès HTTPS forcé et activation SSH

Je clique sur Check for updates dans le tableau de bord pour vérifier les mises à jour disponibles. Ensuite, dans System → Advanced → Admin Access, je m'assure que le protocole de connexion par défaut est bien HTTPS (même si le certificat n'est pas signé par une autorité externe — je traiterai ce point plus loin). Plus bas, j'active le Secure Shell Server (SSH) pour pouvoir prendre la main à distance en ligne de commande de manière sécurisée.

9

Configuration des interfaces et des VLAN (200 serveur, 201 client)

Je gère ensuite les interfaces réseau dans Interfaces → Interface Assignments. Pour la carte LAN, je modifie sa description et je passe en Static IPv4 avec l'IP 192.168.100.250 (déjà configurée précédemment). Je crée ensuite les VLAN qui seront gérés par PFsense en tant que routeur inter-VLAN, dans Interfaces → VLANs → ADD :

  • VLAN_Client : Tag 201, parent interface = LAN.
  • VLAN_Serveur : Tag 200, parent interface = LAN.
Je retourne dans Interfaces → Assignments pour les ajouter en tant qu'interfaces, puis je configure leur IP : 192.168.200.250 pour VLAN_Serveur et 192.168.201.250 pour VLAN_Client. Je n'oublie pas de cocher Enable interface et de sauvegarder pour chacune.

Captures — modification du LAN, création des VLAN 200/201, attribution des IP (8 captures, dans l'ordre du document) :

10

Configuration du DHCP Relay vers les serveurs Beta/Delta

Comme le service DHCP est assuré par les contrôleurs de domaine Beta et Delta (situés dans le VLAN 200), PFsense doit relayer les requêtes DHCP des clients du VLAN 201 vers ces serveurs. Je configure donc le DHCP Relay dans Services → DHCP Relay : j'active le service sur l'interface VLAN_Client et je renseigne les IP des serveurs DHCP cibles (Beta : 192.168.200.201, Delta : 192.168.200.202). Je sauvegarde et le relais DHCP est opérationnel.

Captures — Services → DHCP Relay, configuration et validation (3 captures, dans l'ordre du document) :

11

IDS/IPS — Installation et configuration de Snort

Pour ajouter une couche de détection d'intrusion, j'installe le paquet Snort (IDS/IPS open source) via System → Package Manager → Available Packages. Une fois installé, je vais dans Services → Snort → Global Settings pour la configuration de base. Snort a besoin d'un Oinkcode (clé d'accès aux règles officielles) que j'obtiens en créant un compte sur snort.org et en récupérant la clé dans mon profil. Je colle l'Oinkcode dans la configuration Snort, je lance la mise à jour des règles dans Updates → Update Rules. Ensuite, dans Services → Snort → Interfaces → Add, j'ajoute une instance Snort sur l'interface souhaitée et je la démarre. Les intrusions détectées apparaissent dans l'onglet Alerts.

Captures — installation Snort, oinkcode, mise à jour des règles, ajout d'interface, alertes (8 captures, dans l'ordre du document) :

12

Règles de pare-feu

Je configure les règles de pare-feu dans Firewall → Rules pour autoriser et filtrer le trafic selon les besoins de l'infrastructure : autorisation du trafic sortant vers Internet depuis les VLAN 200 et 201, contrôle du trafic inter-VLAN (les clients VLAN 201 doivent atteindre les services indispensables du VLAN 200 — DHCP relay, DNS, GLPI — mais pas tout le sous-réseau serveur), et j'ajoute également les règles spécifiques nécessaires au futur cluster CARP (synchronisation entre les deux PFsense).

Captures — règles de pare-feu et règles spécifiques au cluster (3 captures, dans l'ordre du document) :

13

Cluster CARP, NAT Hybride, synchronisation HA et sauvegarde XML

Pour la haute disponibilité, je crée le second PFsense Lambda en le configurant à l'identique d'Alpha (sans toucher aux règles, qui se synchroniseront automatiquement). Sur Alpha, je vais dans Firewall → Virtual IPs → ADD pour créer les VIP CARP — une par interface (LAN 192.168.100.1, VLAN 200, VLAN 201) avec le même VHID et le même mot de passe sur les deux PFsense. Ensuite, dans Firewall → NAT → Outbound, je coche Hybrid Outbound NAT rule generation pour utiliser l'adresse VIP LAN comme adresse source par défaut du trafic sortant. Enfin, dans System → High Availability Sync (XMLRPC), j'active la synchronisation entre Alpha (maître) et Lambda (secours) afin que toutes les modifications de configuration (règles, VLAN, NAT, alias) se propagent en temps réel. Je termine par une sauvegarde XML de la configuration via Diagnostics → Backup & Restore, à conserver précieusement pour pouvoir restaurer le pare-feu en cas de défaillance.

Bilan et compétences mobilisées

Ce projet PFsense vient compléter naturellement le Projet 1 (Active Directory) : ensemble, ils forment une infrastructure complète, sécurisée et résiliente. J'ai notamment consolidé les compétences suivantes du référentiel SISR :

  • Installer et configurer un pare-feu open-source (PFsense / FreeBSD) en environnement virtualisé Proxmox.
  • Segmenter un réseau par VLAN 802.1Q (trunks, sous-interfaces, routage inter-VLAN).
  • Mettre en place un relais DHCP pour des serveurs DHCP situés sur un autre VLAN.
  • Définir une politique de filtrage (règles de pare-feu, NAT outbound hybride).
  • Déployer un IDS/IPS (Snort) avec mise à jour automatique des règles via oinkcode.
  • Mettre en place la haute disponibilité avec CARP, VIP partagées et synchronisation XMLRPC.
  • Sauvegarder et restaurer une configuration via un export XML.

Les tests finaux ont validé : un client du VLAN 201 sort sur Internet via le NAT d'Alpha en récupérant une IP DHCP relayée vers Beta, Lambda reprend l'IP CARP 192.168.100.1 en quelques secondes lorsque Alpha est éteinte, la synchronisation XMLRPC est fonctionnelle (toute règle créée sur Alpha apparaît immédiatement sur Lambda) et Snort détecte bien les tentatives d'intrusion simulées.
