Projet 1 — Active Directory, DNS & DHCP

1. Contexte et objectifs

Dans le cadre de mon BTS SIO option SISR, j'ai mené le déploiement complet d'une infrastructure Active Directory destinée à simuler le système d'information d'un établissement scolaire (salle de BTS SIO, professeurs et élèves). Le cahier des charges fixait quatre objectifs :

  • Centraliser la gestion des comptes utilisateurs et des postes via un annuaire AD.
  • Automatiser la distribution des paramètres réseau (IP, passerelle, DNS) avec DHCP.
  • Sécuriser le système d'information par des stratégies de groupe (GPO).
  • Assurer la continuité de service grâce à deux DC et au basculement DHCP.

2. Architecture réseau cible

L'infrastructure globale du projet s'articule autour de deux pare-feu PFsense redondés (traités en détail dans le Projet 2), d'un switch Cisco Theta et de deux VLAN segmentant le trafic serveur et le trafic client. Le schéma ci-dessous récapitule l'ensemble (commun aux deux projets) :

Pare-feu — sortie Internet (LAN BTSSIO 192.168.100.0/24)

  • Alpha — PFsense principal : LAN 192.168.100.250, VLAN 200 192.168.200.250, VLAN 201 192.168.201.250. Rôle : pare-feu, routage NAT, inter-VLAN.
  • Lambda — PFsense de secours : LAN 192.168.100.251. Prend le relais en cas de panne.
  • IP virtuelle (CARP) : 192.168.100.1 — point d'entrée unique côté clients.

Switch Cisco — Theta

  • Ports Fa0/1-12 : VLAN 200 (serveurs).
  • Ports Fa0/13-22 : VLAN 201 (clients).
  • Ports Fa0/23-24 et Gi0/1-2 : trunks vers les PFsense.

VLAN 200 — Serveurs (192.168.200.0/24)

Les serveurs Beta et Delta (en surbrillance) sont au cœur de ce projet.

  • Beta — 192.168.200.201 — AD DS, DNS, DHCP (DC principal).
  • Delta — 192.168.200.202 — AD DS, DNS, DHCP (DC secondaire, DHCP partenaire).
  • Epsilon — 192.168.200.203 — GLPI (gestion de parc).
  • Kappa — 192.168.200.204 — TrueNAS (stockage).
  • Gamma — 192.168.200.211 — Proxmox (hyperviseur hébergeant les VM).

VLAN 201 — Clients (192.168.201.0/24)

  • ETA — borne Wi-Fi en IP fixe : 192.168.201.201 (exclue de la plage DHCP).
  • ZET-01 — poste client, IP 192.168.201.202 attribuée par réservation DHCP sur son adresse MAC.
  • ZET-02 — poste client (IP libre obtenue dans la plage DHCP).
  • Plage DHCP distribuée par Beta (Delta en Hot Standby).
  • Passerelle par défaut : 192.168.201.250 (interface VLAN 201 d'Alpha).

Règles de nommage (norme RFC 1178)

  • Serveurs : un nom unique sous un thème commun (lettres grecques : Alpha, Beta, Gamma, Delta, Epsilon, Theta, Kappa, Lambda).
  • Postes clients : 3 premières lettres du nom de l'entreprise + numéro du PC (ex. ZET-01).
  • Nom de domaine AD : btssio.rd (NetBIOS BTSSIO).

Beta et Delta répliquent automatiquement la base AD. Le DHCP est configuré en mode Hot Standby (actif / passif) : Beta distribue les baux du VLAN 201 et Delta reste en attente. En cas de panne de Beta, Delta prend automatiquement le relais sans interruption pour les clients.

3. Environnement technique

  • Hyperviseur : Proxmox VE (hébergé sur le serveur Gamma — 192.168.200.211) — héberge les VM serveurs.
  • OS serveur (virtualisés sous Proxmox) : Windows Server 2022 Standard — Beta et Delta.
  • OS client (postes physiques, non virtualisés) : Windows 11 Pro.
  • Rôles installés : AD DS, DNS, DHCP.
  • Outils de gestion : Gestionnaire de serveur, Utilisateurs et ordinateurs AD, Gestion DHCP, Gestion DNS, Gestion de stratégie de groupe, Centre d'administration AD.
1

Préparation de Beta et installation des rôles AD DS / DNS / DHCP

Je commence par créer le premier serveur que je nomme Beta. Je lui attribue son adresse IP fixe dans le VLAN 200 (192.168.200.201, masque /24, passerelle 192.168.200.250) et je le mets à jour. Ensuite, depuis Gestionnaire de serveur → Gérer → Ajout de rôles et de fonctionnalités, je sélectionne les trois rôles AD DS, Serveur DNS et Serveur DHCP et je laisse l'assistant installer les binaires.

Promotion de Beta en contrôleur de domaine — création de la forêt btssio.rd

Une fois les rôles installés, je promeus Beta en contrôleur de domaine. Sur le drapeau de notification du Gestionnaire de serveur, je clique sur Promouvoir ce serveur en contrôleur de domaine. Comme c'est une nouvelle installation, je choisis Ajouter une nouvelle forêt et je crée le nom de domaine racine btssio.rd. Je définis le mot de passe DSRM, je laisse cocher Serveur DNS, et je lance la promotion. À la fin, le serveur redémarre et je me connecte directement au domaine.

3

Configuration du DHCP — étendue, exclusion borne Wi-Fi et réservation ZET-01

Je termine la configuration post-déploiement du rôle DHCP via le drapeau, puis je crée une nouvelle étendue dans Outils → DHCP → IPv4 → clic droit → Nouvelle étendue qui couvre toute la plage du VLAN 201 client. Je configure les options standard (passerelle 192.168.201.250, DNS 192.168.200.201, domaine btssio.rd). Comme la borne Wi-Fi ETA sera en IP fixe à 192.168.201.201, j'ajoute une exclusion sur cette adresse pour éviter tout conflit. J'active l'étendue puis je crée une réservation DHCP pour le poste ZET-01 sur sa MAC 74-56-3C-0F-2A-CE à l'IP 192.168.201.202 afin qu'il prenne toujours la même IP.

Test de la configuration DHCP sur ZET-01 — pour valider, je remets l'IP du poste en automatique (en vérifiant que les serveurs DNS sont bien renseignés) puis je lance un ipconfig /all dans une invite de commandes. Toutes les valeurs concordent avec ce qui a été configuré sur Beta : suffixe DNS BTSSIO.RD, adresse IPv4 192.168.201.202 (la réservation par MAC fonctionne), passerelle 192.168.201.250, serveur DHCP 192.168.200.201 (Beta) et serveurs DNS 192.168.200.201 et plus tard 192.168.200.202 (Delta). La distribution DHCP est opérationnelle.

4

Configuration du DNS — zones de recherche inversée et enregistrements PTR

Pour démarrer la configuration DNS, j'ouvre le Gestionnaire DNS depuis Gestionnaire de serveur → Outils → DNS. Le rôle DNS ayant été installé avec AD DS, la zone directe btssio.rd existe déjà. Je crée en plus deux zones de recherche inversée IPv4 : une pour le réseau serveur 192.168.200.0/24 et une pour le réseau client 192.168.201.0/24. Pour chaque zone, je passe par Zones de recherche inversées → clic droit → Nouvelle zone. Une fois les deux zones en place, je crée à la main les pointeurs PTR pour les serveurs et j'active la mise à jour automatique des PTR pour les clients DHCP : DHCP → IPv4 → clic droit → Propriétés → onglet DNS et je coche l'option de mise à jour dynamique.

5

Création des OU, utilisateurs, groupes (GDL) et partages

Dans Outils → Utilisateurs et ordinateurs Active Directory, je crée la structure d'annuaire. Je commence par créer les groupes globaux et des groupes de domaine local (GDL) pour gérer les permissions sur les partages (GDL_Eleves_EL = écriture/lecture, GDL_Eleves_LS = lecture seule, et même chose pour les formateurs). Je crée ensuite les utilisateurs Élèves et Formateurs avec leurs mots de passe. Les formateurs sont membres des deux groupes EL (lecture/écriture sur les deux partages). Les élèves ont l'écriture sur leur partage et seulement la lecture sur celui des formateurs. Je crée également des unités d'organisation (OU) distinctes pour les Élèves et les Formateurs, et je place chaque utilisateur dans la bonne OU pour faciliter l'application des GPO. Je crée enfin une OU pour les ordinateurs de la salle BTSSIO.

6

Ajout du second contrôleur de domaine — Delta

Pour la haute disponibilité, je crée le second serveur Delta, je le renomme, je lui attribue son IP fixe 192.168.200.202 et je fais toutes les mises à jour. Je le joins ensuite au domaine en cliquant sur WORKGROUP et en passant en domaine btssio.rd. Une fois redémarré, j'installe le rôle AD DS et je clique sur le drapeau pour Promouvoir ce serveur en contrôleur de domaine, en choisissant cette fois Ajouter un contrôleur de domaine à un domaine existant. Je saisis le mot de passe DSRM, je sélectionne le DC à répliquer (Beta), j'installe et je redémarre. Pour vérifier que la promotion s'est bien passée, je lance les commandes de diagnostic AD, je vérifie que les zones DNS inversées sont présentes sur Delta et que les utilisateurs/groupes sont répliqués.

Captures — création de Delta, jonction au domaine, promotion en DC secondaire, vérifications (16 captures, dans l'ordre du document) :

7

Configuration du basculement DHCP en mode Hot Standby (Beta ↔ Delta)

Pour rendre le DHCP redondant, je commence par installer le rôle DHCP sur Delta puis je termine sa configuration via le drapeau. Sur Beta, je fais clic droit sur l'étendue → Configurer un basculement. Je désigne Delta comme partenaire et je choisis le mode Serveur de secours (Hot Standby) : Beta reste le serveur actif qui distribue les baux, Delta n'intervient qu'en cas de panne. Je laisse l'intervalle par défaut (1 heure de transition d'état) et je définis un secret partagé entre les deux serveurs. Une fois le basculement créé, je vérifie sur Delta que l'étendue est bien présente. Pour valider, j'éteins volontairement Beta et je m'assure que les postes obtiennent quand même leur IP via Delta.

8

Horaires d'accès des comptes et intégration des postes au domaine

Je gère ensuite les horaires d'accès des comptes utilisateurs : par exemple sur le compte Jayden, je fais clic droit → Propriétés → Compte → Horaires d'accès et je définis l'autorisation du lundi au vendredi. Pour tester, je restreins l'accès jusqu'à 15 h alors qu'il est plus tard, et je vérifie que la connexion est refusée. J'intègre ensuite le poste client au domaine en passant par Propriétés système → Modifier → Domaine btssio.rd. Une fois dans le domaine, le PC apparaît dans Computers dans Active Directory et je peux le glisser dans l'OU des ordinateurs de la salle BTSSIO pour qu'il reçoive les bonnes GPO.

9

GPO de sécurité du domaine — mots de passe forts et options de session

Avant de créer les GPO, je renforce d'abord la politique de mots de passe via Centre d'administration AD → BTSSIO (local) → Système → Password Settings Container (recommandation ANSSI : longueur, complexité, expiration). Je crée ensuite une GPO globale de sécurité que je modifie pour appliquer plusieurs paramètres dans Configuration ordinateur → Stratégies → Paramètres Windows → Paramètres de sécurité → Stratégies locales → Options de sécurité : verrouillage de session, message d'avertissement à la connexion, désactivation du compte invité, audit des connexions, interdiction de stocker les hachages LM et renommage du compte administrateur. Chaque paramètre est testé après application.

GPO utilisateurs — blocage du Panneau de configuration, CMD et PowerShell

Je crée ensuite une GPO destinée aux utilisateurs (élèves notamment) pour limiter ce qu'ils peuvent faire sur les postes. Dans Configuration utilisateur → Stratégies → Modèles d'administration, je configure : Bloquer le Panneau de configuration, Bloquer l'invite de commandes (CMD) via Système → Empêcher l'accès à l'invite de commandes, et Bloquer PowerShell via Système → Ne pas exécuter d'applications Windows en ajoutant powershell.exe et powershell_ise.exe à la liste.

11

GPO ordinateurs — pare-feu, Windows Update, USB, exécution automatique

Pour les ordinateurs, je crée une autre GPO qui durcit la configuration des postes : activation du Pare-feu Windows dans Configuration ordinateur → Stratégies → Paramètres Windows → Paramètres de sécurité → Pare-feu Windows Defender, configuration automatique de Windows Update via Modèles d'administration → Composants Windows → Windows Update (option 4 = téléchargement et planification de l'installation), désactivation du stockage USB (Système → Accès au stockage amovible → Refuser accès en lecture/écriture) et désactivation de l'exécution automatique (Composants Windows → Stratégies d'exécution automatique → Désactiver l'exécution automatique). Tests sur poste : la clé USB n'ouvre plus de dossier automatiquement, le pare-feu est actif, et Windows Update est bien planifié.

12

GPO de déploiement de l'agent GLPI

Pour automatiser l'inventaire de parc, je déploie l'agent GLPI via une GPO. Je copie d'abord le fichier .MSI de l'agent dans un partage déjà présent par défaut sur le DC : \\BTSSIO.RD\NETLOGON. Je modifie ensuite la GPO ordinateur pour ajouter une installation de logiciel attribuée pointant sur le MSI. Pour que l'agent connaisse l'URL du serveur GLPI, j'ajoute une clé de registre via les Préférences → Paramètres Windows → Registre de la même GPO. Sur un poste client, je lance gpupdate /force puis je redémarre : l'agent s'installe et l'ordinateur remonte ensuite automatiquement dans la console GLPI.

Bilan et compétences mobilisées

Ce projet m'a permis de mettre en œuvre de bout en bout une infrastructure Active Directory réaliste, du poste vierge à la console de supervision GLPI. J'ai notamment consolidé les compétences suivantes du référentiel BTS SIO SISR :

  • Installer, configurer et administrer un service (AD DS, DNS, DHCP) sur Windows Server.
  • Assurer la disponibilité d'un service d'annuaire et d'un service DHCP par redondance (Hot Standby).
  • Gérer les identités et les accès via des OU, utilisateurs, groupes globaux et groupes de domaine local (GDL).
  • Définir et déployer des stratégies de sécurité via les GPO (mots de passe, restrictions utilisateurs, durcissement ordinateur).
  • Automatiser le déploiement de logiciels via une GPO d'installation MSI (agent GLPI).
  • Documenter une infrastructure et tracer la configuration en vue d'une reprise d'exploitation.

Les tests finaux ont validé la cohérence de l'ensemble : résolution DNS bidirectionnelle, obtention d'IP via DHCP avec réservation MAC fonctionnelle, authentification AD, application des GPO (utilisateurs et ordinateurs), déploiement automatique de l'agent GLPI, réplication entre Beta et Delta, et basculement automatique lors de l'arrêt volontaire d'un contrôleur.
