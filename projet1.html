Projet 1 — Active Directory, DNS & DHCP

1. Contexte et objectifs

Dans le cadre de mon BTS SIO option SISR, j'ai mené le déploiement complet d'une infrastructure Active Directory destinée à simuler le système d'information d'un établissement scolaire (salle de BTS SIO, professeurs et élèves). Le cahier des charges fixait trois objectifs :

  • Centraliser la gestion des comptes utilisateurs et des postes clients via un annuaire AD.
  • Automatiser la distribution des paramètres réseau (IP, passerelle, DNS) avec un serveur DHCP.
  • Assurer la continuité de service grâce à deux contrôleurs de domaine et un basculement DHCP.

Le projet couvre l'ensemble du cycle : installation, configuration, mise en production, tests et documentation. Il mobilise directement les compétences du référentiel SISR en administration des systèmes et des réseaux.

2. Architecture réseau cible

L'infrastructure globale du projet s'articule autour de deux pare-feu PFsense redondés (traités en détail dans le Projet 2), d'un switch Cisco Theta et de deux VLAN segmentant le trafic serveur et le trafic client. Le schéma ci-dessous récapitule l'ensemble (commun aux deux projets) :

Pare-feu — sortie Internet (LAN BTSSIO 192.168.100.0/24)

  • Alpha — PFsense principal : LAN 192.168.100.250, VLAN 200 192.168.200.250, VLAN 201 192.168.201.250. Rôle : pare-feu, routage NAT, inter-VLAN.
  • Lambda — PFsense de secours : LAN 192.168.100.251. Prend le relais automatiquement en cas de panne.
  • IP virtuelle (CARP) : 192.168.100.1 — point d'entrée unique vue des clients.

Switch Cisco — Theta

  • Ports Fa0/1-12 : VLAN 200 (serveurs).
  • Ports Fa0/13-22 : VLAN 201 (clients).
  • Ports Fa0/23-24 et Gi0/1-2 : trunks vers les PFsense.

VLAN 200 — Serveurs (192.168.200.0/24)

  • Beta — 192.168.200.201 — AD DS, DNS, DHCP (DC principal) — périmètre de ce projet.
  • Delta — 192.168.200.202 — AD DS, DNS, DHCP (DC secondaire, DHCP partenaire) — périmètre de ce projet.
  • Epsilon — 192.168.200.203 — GLPI (gestion de parc).
  • Kappa — 192.168.200.204 — TrueNAS (stockage).
  • Gamma — 192.168.200.211 — Proxmox (hyperviseur hébergeant les VM).

VLAN 201 — Clients (192.168.201.0/24)

  • ETA — borne Wi-Fi en IP fixe : 192.168.201.201 (exclue de la plage DHCP).
  • ZET-01 — poste client, IP 192.168.201.202 attribuée par réservation DHCP sur son adresse MAC.
  • ZET-02 — poste client (IP libre obtenue dans la plage DHCP).
  • Plage DHCP distribuée par Beta/Delta : 192.168.201.10 → 192.168.201.249.
  • Exclusion DHCP : 192.168.201.201 (réservée à la borne Wi-Fi en IP fixe).
  • Réservations DHCP par MAC : 192.168.201.202 = ZET-01.
  • Passerelle par défaut : 192.168.201.250 (interface VLAN 201 d'Alpha).

Règles de nommage (norme RFC 1178)

  • Serveurs : un nom unique, généralement sous un même thème (lettres grecques dans notre cas : Alpha, Beta, Gamma, Delta, Epsilon, Theta, Kappa, Lambda).
  • Postes clients : 3 premières lettres du nom de l'entreprise + numéro du PC (ex. ZET-01).
  • Nom de domaine AD : btssio.rd (NetBIOS BTSSIO).

Les deux contrôleurs Beta et Delta répliquent automatiquement la base AD. Le DHCP fonctionne en mode équilibrage de charge (50 / 50) : chaque serveur distribue la moitié des baux du VLAN 201 ; en cas de panne, l'autre prend 100 % de la charge sans interruption pour les clients.

3. Environnement technique

  • Hyperviseur : VMware Workstation (machines virtuelles isolées sur un même vSwitch).
  • OS serveur : Windows Server 2022 Standard (GUI) — 2 machines.
  • OS client de test : Windows 10 Pro.
  • Rôles installés : AD DS, DNS, DHCP.
  • Outils de gestion : Gestionnaire de serveur, Utilisateurs et ordinateurs AD, Gestion DHCP, Gestion DNS, Gestion de stratégie de groupe.
1

Préparation du premier serveur (Beta)

Avant d'installer le moindre rôle, il faut préparer la machine qui deviendra le contrôleur de domaine principal. Je renomme le serveur en Beta via Gestionnaire de serveur, je lui attribue une adresse IP fixe dans le VLAN 200 (serveurs) — 192.168.200.201, masque /24, passerelle 192.168.200.250 (interface VLAN 200 d'Alpha) — et je fixe son serveur DNS préféré sur lui-même (127.0.0.1). Cette étape est critique : un DC change très mal d'adresse ou de nom une fois promu.

Installation du rôle AD DS, DNS et DHCP sur Beta

Depuis Gestionnaire de serveur → « Ajouter des rôles et fonctionnalités », je sélectionne successivement Services AD DS, Serveur DNS et Serveur DHCP. L'assistant installe les binaires mais ne promeut pas encore la machine : c'est une étape séparée. Je vérifie en fin d'installation que les trois rôles apparaissent bien dans le tableau de bord.

3

Promotion de Beta et création de la forêt btssio.rd

L'icône de notification du Gestionnaire de serveur propose « Promouvoir ce serveur en contrôleur de domaine ». Je choisis Ajouter une nouvelle forêt avec le nom de domaine racine btssio.rd. Je laisse les niveaux fonctionnels en Windows Server 2016 (par défaut), je coche bien Serveur DNS et je définis le mot de passe DSRM (restauration des services d'annuaire). Le nom NetBIOS BTSSIO est généré automatiquement. L'assistant effectue les vérifications de prérequis puis redémarre la machine : Beta est désormais contrôleur de domaine de btssio.rd.

Captures — assistant de promotion, choix de la forêt, niveaux fonctionnels, prérequis (18 captures) :

4

Installation et configuration du serveur DHCP (étendue principale)

Après la promotion, je termine la configuration post-déploiement du rôle DHCP (autorisation dans l'AD avec un compte de l'administrateur du domaine). Le DHCP est hébergé dans le VLAN 200 (sur Beta) mais distribue les baux aux postes du VLAN 201 (clients) grâce au relais inter-VLAN assuré par PFsense. Je crée donc une nouvelle étendue couvrant 192.168.201.10 → 192.168.201.249 (masque /24). Je renseigne les options standard : passerelle par défaut 192.168.201.250 (interface VLAN 201 d'Alpha), serveur DNS 192.168.200.201 (Beta) et nom de domaine btssio.rd. J'active l'étendue à la fin de l'assistant.

Pour intégrer la borne Wi-Fi ETA en IP fixe (192.168.201.201) sans risquer de conflit, j'ajoute une exclusion de cette adresse dans l'étendue (assistant Ajout d'exclusions et de retard, IP de début = 192.168.201.201, IP de fin laissée vide = exclusion d'une seule adresse). Je crée également une réservation DHCP pour le poste ZET-01 liée à son adresse MAC 74-56-3C-0F-2A-CE avec l'IP 192.168.201.202 et le nom ZET-01.BTSSIO.RD (type pris en charge : « Les deux »).

Captures — création de l'étendue DHCP, plage d'adresses, options de l'étendue (14 captures) :

5

Configuration du DNS et zones de recherche inversée

Le DNS a été installé en même temps qu'AD DS, une zone directe btssio.rd existe donc déjà. Je crée en plus deux zones de recherche inversée IPv4 — une pour le VLAN 200 (192.168.200.0/24) et une pour le VLAN 201 (192.168.201.0/24) — afin que les requêtes « PTR » (IP → nom) fonctionnent dans les deux sous-réseaux. J'ajoute quelques enregistrements A et PTR manuels pour valider le bon fonctionnement et je teste la résolution avec nslookup depuis un client du VLAN 201.

Captures — gestionnaire DNS, assistant de zone inversée, tests de résolution (14 captures) :

6

Création des OU, utilisateurs et groupes Active Directory

Dans la console Utilisateurs et ordinateurs Active Directory, je crée une arborescence d'unités d'organisation représentative de l'établissement : Salle_BTSSIO → Professeurs et Eleves, puis des utilisateurs dans chacune. Je crée également des groupes de sécurité (Groupe_Professeurs, Groupe_Eleves) et j'y ajoute les membres correspondants. Chaque utilisateur reçoit un mot de passe initial avec l'option « changement au prochain ouverture de session » pour respecter les bonnes pratiques de sécurité.

Captures — création des OU, ajout des utilisateurs, affectation aux groupes (20 captures) :

7

Ajout d'un second contrôleur de domaine (Delta)

Pour la haute disponibilité, j'installe un second serveur, Delta, dans le VLAN 200 avec l'IP fixe 192.168.200.202 (passerelle 192.168.200.250, DNS préféré 192.168.200.201 qui pointe vers Beta). Je le joins au domaine btssio.rd puis je l'élève en DC supplémentaire via l'assistant « Promouvoir ce serveur en contrôleur de domaine » → Ajouter un contrôleur de domaine à un domaine existant. Je coche DNS et catalogue global de manière à ce que Delta puisse assurer toutes les fonctions en cas de panne de Beta. La réplication AD se met en place automatiquement entre les deux DC.

Captures — jonction au domaine, promotion de Delta, réplication AD (20 captures) :

8

Configuration du basculement DHCP (DHCP Failover)

Sur la console DHCP de Beta, un clic droit sur l'étendue → « Configurer un basculement » permet de mettre en place la redondance DHCP avec Delta comme partenaire. Je choisis le mode Équilibrage de charge (50 % / 50 %) avec un secret partagé. À la fin de l'assistant, Delta récupère automatiquement la configuration de l'étendue et les deux serveurs se synchronisent. Si l'un tombe, l'autre continue d'attribuer les baux sans interruption pour les postes clients.

Captures — assistant de basculement, mode d'équilibrage, vérification sur Delta (14 captures) :

9

Mise en place des stratégies de groupe (GPO)

Dans Gestion de stratégie de groupe, je crée des GPO liées aux OU pour appliquer des restrictions et des paramètres : désactivation de l'exécution automatique des supports amovibles, redirection des dossiers, stratégies de mot de passe, fond d'écran verrouillé pour les élèves, refus d'accès au Panneau de configuration, etc. Chaque GPO est d'abord créée, puis éditée (Configuration ordinateur ou utilisateur) et enfin liée à l'OU concernée. Un gpupdate /force sur un poste client permet de vérifier que les paramètres s'appliquent bien.

Captures — création et édition des GPO, liaisons aux OU, paramètres de sécurité (21 captures) :

10

Tests et vérifications finaux

Pour valider l'infrastructure, j'effectue une série de tests depuis un poste client situé dans le VLAN 201 : jonction du poste Windows 10 au domaine btssio.rd, ouverture de session avec un utilisateur AD, vérification que les GPO s'appliquent, obtention d'une adresse IP via DHCP. Une commande ipconfig /all sur le poste ZET-01 confirme la cohérence complète de la configuration : suffixe DNS BTSSIO.RD, adresse IPv4 192.168.201.202 (la réservation DHCP est bien appliquée à la MAC 74-56-3C-0F-2A-CE), passerelle 192.168.201.250, serveur DHCP 192.168.200.201 (Beta) et serveurs DNS 192.168.200.201 + 192.168.200.202 (Beta + Delta). Je teste aussi la résolution DNS directe et inverse (nslookup beta.btssio.rd, nslookup 192.168.200.201), j'éteins volontairement Beta pour vérifier que Delta prend le relais (DHCP + authentification) sans interruption pour les clients, puis je contrôle la réplication AD avec repadmin /replsummary.

Bilan et compétences mobilisées

Ce projet m'a permis de mettre en œuvre de bout en bout une infrastructure Active Directory réaliste, du poste vierge à la console de supervision. J'ai notamment consolidé les compétences suivantes, directement rattachées au référentiel BTS SIO SISR :

  • Installer, configurer et administrer un service (AD DS, DNS, DHCP) sur Windows Server.
  • Assurer la disponibilité d'un service d'annuaire et d'un service DHCP par redondance.
  • Gérer les identités et les accès via des OU, utilisateurs et groupes AD.
  • Déployer et centraliser les paramètres via les stratégies de groupe (GPO).
  • Documenter une infrastructure et tracer la configuration en vue d'une reprise d'exploitation.

Les tests finaux ont validé la cohérence de l'ensemble : résolution DNS bidirectionnelle, obtention d'IP via DHCP, authentification AD, application des GPO, réplication entre Beta et Delta et basculement automatique lors de l'arrêt volontaire d'un contrôleur.
